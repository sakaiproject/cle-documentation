1.  General Guidelines

The question of how to edit and deploy the documentating being developed here has not been addressed at this time.  Ideally, the development process could support multiple tools, which would allow each author to decide for themselves.  However, that could also lead to chaos.  Rather than picking a tool, this document describes some guildlines that should be followed to simplify the authoring, management, and deployment process later (to be defined).

1.1  Markup - XHTML

All documents should be marked up using XHTML.  While this does add a bit more work, the advantages are considerable.  Using XHTML allows documents created to be parsed using XML tools such as SAX.  Parsable documents can be later processed by (TBD) scripts to create deployable forms of the documentation.

1.2  Layout - Numbered Sections

Documents should be sectioned using standard HTML section tags:  H1, H2, etc.  Documents may have more than one top level section.  Good use of sectioning improves accessibility.  Numbering provides support for referencing.

1.3  Styling

In it's raw form, technical documentation should reference any style sheet.  Eventually, we might create one for convenience or provide a way to preview different kinds of style.  Until we do, don't bother with global styling.  Text styling for emphasis is a different matter.  Embedded style can be used appropriately.  More detailed guidelines on this will be developed at a later date.

1.4  Use of Illustrations and Examples

The use of illustrations in technical documents is strongly encouraged. All illustrations should be stored in a directory called "images".  Image formats should be restricted to JPEG or PNG, with PNG preferred.

1.4.1  Screen Shots

Screen shots should be cropped to show the specific area of interest.  Screen shot images should be no wider than 800 pixels.  If a larger image is needed, it should be scaled down to 800 pixels for display.  Most browsers support clicking on the image to enlarge it.

1.4.2  Graphics

Artistic illustrations, charts, diagrams, etc. are subject to the same restrictions above.

1.4.3  Code Samples

Code samples should be formatted using the <PRE> tag.

1.5  Authors

At the bottom of each page, include an element of the following form:

    <div id="author">Mark Norton, Glen Golden</div>

The primary author should be listed first, followed by other contributors, even editors.  This special element allows authors to be identified and credited.

1.6  Sakai Versioning

At the bottom of the page, include an element of the following form:

    <div id="version">2.9</div>

The  first Sakai version that this text applies to should be identified.  If changes to Sakai require a change in the technical documentation, enter the most recent verison number instead.  Source control will allow us to create tagged doc sets for older versions of Sakai using this technique.  If the verision is unknown, use "0.0".
